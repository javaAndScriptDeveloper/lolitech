# GitOps with FluxCD — Lab Work

FluxCD v2.2.3 cluster configuration demonstrating the GitOps model: the repository is the source of truth and Flux continuously reconciles the live Kubernetes cluster state against what is committed here.

**Stack:** FluxCD v2 · Kubernetes · Kustomize · GitOps

---

## How it works

Flux runs four controllers inside the cluster (`flux-system` namespace):

| Controller | Role |
|------------|------|
| `source-controller` | Watches Git repositories and Helm repos for changes |
| `kustomize-controller` | Applies Kustomize overlays from source artifacts |
| `helm-controller` | Manages Helm releases |
| `notification-controller` | Sends alerts on reconciliation events |

When a commit lands in this repo, `source-controller` detects it within 1 minute. `kustomize-controller` then applies the updated manifests to the cluster. No `kubectl apply` is ever run manually.

---

## Repository structure

**`flux-system/gotk-components.yaml`** — the full set of Flux CRDs, RBAC rules, network policies, and controller Deployments. Generated by `flux install` and checked in so the cluster can be bootstrapped from git alone.

**`flux-system/gotk-sync.yaml`** — two resources that close the GitOps loop:
- A `GitRepository` pointing to this repo (branch `main`, polled every 1 minute).
- A `Kustomization` telling Flux to apply everything under `./` from that source, with `prune: true` so resources deleted from git are also deleted from the cluster.

**`flux-system/kustomization.yaml`** — Kustomize entry point that pulls in both `gotk-components.yaml` and `gotk-sync.yaml`.

## Example app: podinfo

`podinfo-source.yaml` and `podinfo-kustomization.yaml` show how to track an external application:

- A `GitRepository` watches `stefanprodan/podinfo` on branch `master`, polling every 1 minute.
- A `Kustomization` deploys it to the `default` namespace from the `./kustomize` path in that repo, reconciling every 30 minutes with a 3-minute timeout and automatic retry on failure.

This demonstrates the typical GitOps pattern of separating the config repo (this one) from the app repo (podinfo), with Flux bridging them.
